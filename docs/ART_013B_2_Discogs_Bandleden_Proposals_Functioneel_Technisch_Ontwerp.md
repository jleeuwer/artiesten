# ART-013B-2 — Discogs ondersteuning en reviewqueue voor bandledenrelaties

Datum: 2026-07-12  
Status: functioneel en technisch ontwerp voor volgende codesprint

## 1. Doel

ART-013B-2 ondersteunt gebruikers bij het aanvullen van bandledeninformatie wanneer zij niet alle leden, rollen of brongegevens kennen.

Uitgangspunten:

1. Handmatige invoer blijft altijd mogelijk en leidend.
2. `musician_in_band` blijft de lokale waarheid.
3. Discogs is de primaire externe bron.
4. Discogs schrijft nooit rechtstreeks naar `musician_in_band`.
5. Resultaten komen eerst in een reviewqueue.
6. Acceptatie is altijd een expliciete gebruikersactie.
7. MusicBrainz vult in een latere sprint aan.
8. Wikidata volgt pas daarna als aanvullende verificatiebron.
9. Een voorgesteld bandlid hoeft niet als artist te bestaan.
10. Een nieuw voorgesteld bandlid kan als standalone musician worden aangemaakt.

## 2. Functionele gebruikersflow

### 2.1 Startpunt

Vanuit het onderdeel **Bandleden** bij een band kan de gebruiker kiezen:

- Bandlid handmatig toevoegen;
- Zoek aanvullende informatie via Discogs;
- Open reviewqueue.

Handmatig toevoegen blijft onafhankelijk van Discogs beschikbaar.

### 2.2 Discogs ophalen

De gebruiker start een Discogs-zoekactie voor de geselecteerde band.

De applicatie gebruikt bij voorkeur de reeds gekoppelde Discogs-artist van de band. Als geen koppeling bestaat, wordt eerst de bestaande Discogs zoek- en koppelflow gebruikt.

Discogs kan onder meer leveren:

- leden van de groep;
- voormalige leden;
- naamvarianten;
- Discogs artist-id;
- profiel- of bron-URL;
- groepen waarvan een persoon lid is;
- beperkte rol- of creditinformatie.

Ontbrekende periode- of rolinformatie is toegestaan.

### 2.3 Reviewqueue

Ieder gevonden kandidaat-lid komt als voorstel in de reviewqueue. De gebruiker ziet:

- voorgestelde naam;
- Discogs-id en bronlink;
- mogelijke lokale musicianmatch;
- gekoppelde artist, indien aanwezig;
- bestaande bandrelatie, indien aanwezig;
- voorgestelde rol;
- voorgestelde periode;
- matchstatus;
- voorstelstatus;
- conflict- of aandachtspunt.

### 2.4 Mogelijke uitkomsten

Een voorstel kan worden geclassificeerd als:

- `new_musician`: geen lokale musician gevonden;
- `matched_musician`: lokale musician gevonden, relatie ontbreekt;
- `matched_relation`: relatie bestaat al en komt overeen;
- `partial`: relatie bestaat, maar Discogs bevat aanvullende gegevens;
- `conflict`: lokale en externe gegevens spreken elkaar tegen;
- `ambiguous`: meerdere mogelijke lokale matches;
- `invalid`: onvoldoende bruikbare brondata.

### 2.5 Gebruikersacties

Per voorstel:

- **Koppel bestaande musician**;
- **Maak standalone musician en voeg toe**;
- **Vul bestaande relatie aan**;
- **Alleen als bron koppelen**;
- **Later**;
- **Negeer**;
- **Heropen**.

Geen enkele actie wijzigt de canonical artistnaam.

### 2.6 Acceptatie van nieuwe musician

Wanneer geen lokale musician bestaat:

1. gebruiker controleert naam en eventuele datums;
2. applicatie toont mogelijke duplicaten;
3. gebruiker bevestigt;
4. transactioneel wordt een standalone musician aangemaakt;
5. transactioneel wordt `musician_in_band` aangemaakt;
6. Discogs-bron wordt aan de relatie gekoppeld;
7. proposalstatus wordt `accepted`.

Er wordt niet automatisch een artist aangemaakt.

### 2.7 Acceptatie van bestaande musician

Wanneer een match bestaat:

1. gebruiker bevestigt de juiste musician;
2. relatie wordt aangemaakt of aangevuld;
3. lokale handmatige waarden worden niet stil overschreven;
4. verschillen worden expliciet getoond;
5. gebruiker kiest per veld welke waarde wordt toegepast.

### 2.8 Bestaande relatie

Als de relatie al bestaat:

- gelijke brondata wordt als `matched_relation` gemarkeerd;
- aanvullende brondata wordt `partial`;
- afwijkende rol of periode wordt `conflict`;
- bron kan zonder datamutatie worden toegevoegd.

### 2.9 Idempotentie

Het opnieuw ophalen van dezelfde Discogs-data:

- maakt geen dubbele voorstellen;
- maakt geen dubbele sources;
- bewaart `accepted`, `ignored` en `review_later`;
- actualiseert bronpayload en `retrieved_at`;
- herclassificeert alleen wanneer lokale data aantoonbaar is veranderd.

## 3. Datamodel

### 3.1 Bestaande lokale waarheid

`musician_in_band` blijft ongewijzigd de definitieve lokale relatie.

### 3.2 Nieuwe proposal-tabel

```sql
CREATE TABLE public.musician_in_band_proposal (
  mibp_key bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  band_artist_key bigint NOT NULL,
  proposed_musician_key bigint NULL,
  proposed_relation_key bigint NULL,
  source_type text NOT NULL DEFAULT 'discogs',
  source_band_id text NOT NULL,
  source_person_id text NOT NULL,
  source_relationship_id text NULL,
  proposed_person_name text NOT NULL,
  proposed_role text NULL,
  proposed_date_from date NULL,
  proposed_date_to date NULL,
  proposed_date_from_precision text NULL,
  proposed_date_to_precision text NULL,
  match_status text NOT NULL,
  status text NOT NULL DEFAULT 'new',
  confidence numeric(5,2) NULL,
  conflict_code text NULL,
  conflict_details jsonb NULL,
  raw_payload jsonb NOT NULL,
  retrieved_at timestamptz NOT NULL DEFAULT now(),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

Voorstelstatussen:

- `new`
- `review_later`
- `accepted`
- `ignored`
- `conflict`
- `invalid`

### 3.3 Nieuwe brontabel

```sql
CREATE TABLE public.musician_in_band_source (
  mibs_key bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  musician_in_band_key bigint NOT NULL,
  source_type text NOT NULL,
  source_entity_id text NULL,
  source_relationship_id text NULL,
  source_url text NULL,
  source_role text NULL,
  source_date_from date NULL,
  source_date_to date NULL,
  source_payload jsonb NULL,
  retrieved_at timestamptz NOT NULL DEFAULT now(),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

De brontabel is generiek zodat later `musicbrainz` en `wikidata` dezelfde structuur gebruiken.

### 3.4 Unieke sleutels

Voor proposals:

```text
band_artist_key + source_type + source_band_id + source_person_id + source_relationship_id
```

Voor sources:

```text
musician_in_band_key + source_type + source_entity_id + source_relationship_id
```

NULL-bronrelatie-id's moeten met een expression index of genormaliseerde lege waarde idempotent worden behandeld.

### 3.5 Foreign keys

- proposal.band_artist_key → artist;
- proposal.proposed_musician_key → musician, nullable;
- proposal.proposed_relation_key → musician_in_band, nullable;
- source.musician_in_band_key → musician_in_band.

Verwijderen van een lokale relatie mag sources niet stil orphanen. Keuze: restrict of transactioneel meeverwijderen met auditmelding. Voorkeur: transactioneel verwijderen van sources bij expliciete relatie-delete.

## 4. Matchingstrategie

Matchen gebeurt in lagen:

1. bestaande externe Discogs-koppeling op musician/artist;
2. exact genormaliseerde naam;
3. alternatieve spelling;
4. naam plus geboorte-/sterfdatum;
5. bestaande bandrelatie;
6. meerdere kandidaten → `ambiguous`.

Automatische acceptatie is niet toegestaan.

Confidence ondersteunt sortering en uitleg, niet automatische toepassing.

## 5. Bronprioriteit en veldregels

Prioriteit:

1. lokaal handmatig bevestigd;
2. Discogs als primaire voorstelbron;
3. MusicBrainz later als aanvullende bron;
4. Wikidata later als verificatie.

Een bestaande lokale waarde wordt nooit automatisch vervangen. Bij een leeg lokaal veld mag de UI voorstellen **Vul leeg veld aan**. Bij verschil toont de UI beide waarden en vraagt expliciete keuze.

## 8. Transactionele acceptatie

### Nieuwe musician

```text
BEGIN
lock proposal
validate status and expectedUpdatedAt
recheck duplicates and conflicts
create standalone musician
create musician_in_band
create musician_in_band_source
mark proposal accepted
COMMIT
```

### Bestaande musician

```text
BEGIN
lock proposal and musician
recheck relation
create or update musician_in_band
create source idempotently
mark proposal accepted
COMMIT
```

### Alleen bron koppelen

```text
BEGIN
lock proposal and relation
create source idempotently
mark proposal accepted
COMMIT
```

Elke fout leidt tot rollback.

## 13. Acceptatiecriteria

1. Handmatige invoer blijft volledig werken zonder Discogs.
2. Alleen gekoppelde bands kunnen Discogs-proposals genereren.
3. Genereren is idempotent.
4. Geen proposal schrijft direct naar `musician_in_band`.
5. Nieuwe standalone musician kan expliciet worden geaccepteerd.
6. Bestaande musician kan expliciet worden gekoppeld.
7. Bestaande relaties kunnen alleen expliciet worden aangevuld.
8. Lokale waarden worden nooit stil overschreven.
9. Iedere geaccepteerde relatie kan één of meer bronnen hebben.
10. Stale en gelijktijdige acties worden veilig geweigerd.
11. Alle mutaties zijn transactioneel.
12. Database-integratietests eindigen met `leftovers=0`.
13. Bestaande ART-013A-3- en ART-013B-1-regressies blijven groen.

## Fix-5 — Artist-aware matching en artist-type diagnostiek

Voor ieder Discogs-bandlid zoekt de applicatie eerst naar musicians en daarna naar artists via Discogs-ID, canonical naam en alternatieve spelling. Een bestaande artistmatch wordt nooit genegeerd omdat `ar_artist_type` leeg, `unknown` of foutief is.

- `person`: direct gebruiken of musician aanmaken en koppelen.
- leeg/`unknown`: datakwaliteitswaarschuwing en blokkade.
- ander type: typeconflict en blokkade.
- pas zonder artist- of musicianmatch is `new_musician` toegestaan.
